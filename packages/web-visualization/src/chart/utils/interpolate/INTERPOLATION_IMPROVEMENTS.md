# Finance Chart Path Interpolation -- Test Scenarios

## Background

We use SVG path interpolation to animate between chart states (e.g., time range changes, new data points arriving, period switches). The base algorithm is adapted from `d3-interpolate-path`: it parses two SVG `d` strings into command arrays, extends the shorter one to match the longer one by splitting Bezier segments, then linearly interpolates all numeric parameters at each `t` value.

This works well for simple cases but produces poor animations for several finance-chart-specific scenarios. Below are 7 test cases with detailed analysis.

## Proposed Approach

Before extending/splitting, compare the command arrays from A and B to find segments that are structurally the same (matching values within tolerance). Any matching segments -- whether at the start, end, or middle of the path -- should be paired directly and not split. Only the non-matching regions should go through extension/splitting. This single generic mechanism should handle all the scenarios below without scenario-specific detection.

## Test 1 -- Slide Data

**Scenario:** Real-time data streaming -- a new data point enters from the right while the oldest drops off the left. The Y values shift by one position: `B[i].y = A[i+1].y`.

**Start (6 data points, x-grid: 0, 60, 120, 180, 240, 300):**
```
M0,180 C30,180,30,140,60,140 C90,140,90,100,120,100 C150,100,150,60,180,60 C210,60,210,20,240,20 C270,20,270,60,300,60
```

**End (6 data points, same x-grid, Y shifted by 1):**
```
M0,140 C30,140,30,100,60,100 C90,100,90,60,120,60 C150,60,150,20,180,20 C210,20,210,60,240,60 C270,60,270,40,300,40
```

**Expected at t=0.5 (7 commands -- one MORE than the inputs):**
```
M0,160 C7.5,150,15,140,30,140 C60,140,60,100,90,100 C120,100,120,60,150,60 C180,60,180,20,210,20 C240,20,240,60,270,60 C285,60,292.5,55,300,50
```

**What the expected output shows:**
- The output has n+1 commands (7 vs 6). This is a sliding window.
- The first segment is a HALF curve (x goes from 0 to 30, not 0 to 60) -- the "entering" portion.
- The middle segments (indices 2-5) are at the B positions.
- The last segment is a HALF curve (x from 270 to 300) -- the "exiting" portion.
- The half-curves are generated by de Casteljau splitting the first/last segments of A at `(1-t)` and `t` respectively, then x-shifting to align.

## Test 2 -- Time Change (Different X Grids, Different Counts)

**Scenario:** User switches from one time period to another. Both A and B have different numbers of data points AND different X grids. The path structures are unrelated.

**Start (14 data points, x: 16 to 485, step ~18):**
```
M16,168.8 C34.038,168.8,...,485,95.84
```

**End (17 data points, x: 16 to 485, step ~14.7):**
```
M16,124.72 C30.656,124.72,...,485,155.12
```

**Expected at t=0.5:** Commands at evenly-spaced X positions between A's and B's grids, with Y values that are midpoints. The expected output has 17 commands with X values like `32.347, 48.695, 65.042, ...` -- which are the midpoints of A's and B's X grids.

**Key challenge:** When extending A (14 commands) to match B (17 commands), new points are placed evenly by curve parameter rather than at B's X positions.

## Test 3 -- Simple to Complex Area (Time Change)

**Scenario:** A 1-week view (2 data points = nearly flat line) transitions to a 1-year view (575+ data points = detailed curve). This is an area path.

**Start (rectangle area -- M, C, L, bottom-C, Z):**
```
M16,134.5 C410.5,134.5,410.5,134.5,805,134.5 L805,209 C410.5,209,410.5,209,16,209 Z
```

**End (detailed area -- M, 575 C commands, L, 575 bottom-C commands, Z):**
```
M16,100.276 C17.375,100.276,...,805,101.274 L805,3770.1 C803.625,3770.1,...,16,3770.1 Z
```

**Expected at t=0.5:** Each command in the output should be at the EXACT X position of B's command, with Y as the midpoint between A's flat value and B's data value.

**Key structural insight:** The area path has 4 distinct sections:
1. **Top edge** (M + C commands, x goes left-to-right) -- this is the DATA
2. **Right side** (L command, goes from top-right to bottom-right)
3. **Bottom edge** (C commands, x goes right-to-left at baseline y) -- this is NOT data
4. **Close** (Z)

The L, bottom edge, and Z are structurally similar between A and B (same X positions, flat Y). These matching segments should be identified and paired directly. Only the top edge (which differs in point count) should be split/extended.

## Test 4 -- Simple to Complex Line (Time Change)

**Scenario:** Same as Test 3 but for a line path (no area bottom edge). A 1-week view line transitions to a 1-year line.

**Start (1 segment):**
```
M16,134.5 C410.5,134.5,410.5,134.5,805,134.5
```

**End (575 segments):**
```
M16,100.276 C17.375,100.276,...,805,101.274
```

**Expected at t=0.5:** Commands at B's exact X positions with Y as midpoints.

**Key challenge:** The single C segment needs to be split into 574 segments with points landing at B's specific X positions, not evenly by curve parameter.

---

## Test 5 -- Line Extension

**Scenario:** A new data point is appended to the end of a line chart.

**Start (5 points):**
```
M16,69.929 C35.605,69.929,...,172.842,75.285
```

**End (6 points):**
```
M16,69.929 C35.605,69.929,...,172.842,75.285 C192.447,75.285,...,212.053,116.387
```

**Expected at t=0.5:** The first 5 points stay fixed. Only the new 6th point animates in from the 5th point's location.

**Key insight:** B starts with all of A's commands -- they are structurally identical. These matching segments should be identified and locked. Only the new trailing segment should animate.

---

## Test 6 -- Line with Gap (Vertical Change)

**Scenario:** Two separate line segments (with a gap between them) both shift vertically.

**Start:**
```
M16,32 C78.083,32,...,140.167,97.909
M388.5,112.409 C450.583,112.409,...,761,65.614
```

**End:**
```
M16,111.091 C78.083,111.091,...,140.167,157.227
M388.5,69.964 C450.583,69.964,...,761,32.527
```

**Expected at t=0.5:** Each line segment morphs vertically independently. The gap is preserved.

**Key insight:** The path contains multiple `M` commands indicating separate sub-paths. Each sub-path should be interpolated independently.

---

## Test 7 -- Area with Gap (Vertical Change)

**Scenario:** Same as Test 6, but for area paths (which include L, bottom edges, and Z).

**Same approach as Test 6** -- sub-path splitting handles this since each area sub-path (with its own Z) gets interpolated independently.
